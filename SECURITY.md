# Security Policy

## Supported Scope

Security reports are welcome for issues that affect:

- sandboxed file access
- metadata persistence
- secret or private content exposure
- unsafe external file handling
- dependency or build configuration issues with clear security impact

## Reporting a Vulnerability

Please do not report security vulnerabilities in public issues.

Instead:

1. Prepare a private report with reproduction details, impact, and affected files.
2. Send it to the project maintainers through a non-public channel.
3. If the repository is hosted on GitHub and private vulnerability reporting is enabled, prefer that path.
4. Include any suggested mitigation if you have one.

## What to Include

Please include:

- a clear description of the issue
- attack preconditions
- reproduction steps
- affected files or components
- impact assessment
- whether private data or local paths may be exposed

## Response Expectations

Best effort process:

- acknowledge receipt
- evaluate severity and reproducibility
- prepare a fix or mitigation
- coordinate disclosure timing if public release is appropriate

No SLA is guaranteed in the current repository state.

## Safe Harbor

If you act in good faith, avoid privacy violations, avoid service disruption,
and do not exploit the issue beyond what is necessary to demonstrate it, your
report will be treated as responsible disclosure.
